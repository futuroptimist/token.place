API v1 launch contract (0.1.x)

API v1 is the frozen launch surface for token.place 0.1.0/0.1.x. It is non-streaming: stream: true is rejected on API v1 completion routes. API v2 is intentionally not part of this launch contract.

The /v1 OpenAI-compatible aliases intentionally mirror the core OpenAI-style API routes: /v1/models, /v1/models/{model_id}, /v1/public-key, /v1/public-key/rotate, /v1/chat/completions, /v1/completions, /v1/images/generations, /v1/health, and /v1/relay/unregister.

Public and client-facing API v1 routes

GET /api/v1/models

Lists the available launch API v1 models in OpenAI-compatible list format. The catalog is curated for API v1 and is not an API v2 capability dump.

Example Response:

{
  "object": "list",
  "data": [
    {
      "id": "llama-3-8b-instruct",
      "object": "model",
      "created": 1700000000,
      "owned_by": "token.place",
      "permission": [
        {
          "object": "model_permission",
          "allow_sampling": true
        }
      ],
      "root": "llama-3-8b-instruct",
      "parent": null
    }
  ]
}

GET /api/v1/models/{model_id}

Retrieves one model from the API v1 model list using the same OpenAI-compatible model object shape.

GET /api/v1/public-key

Returns the server public key that clients use before sending encrypted requests.

{
  "public_key": "BASE64_PUBLIC_KEY_HERE"
}

POST /api/v1/public-key/rotate

Operator-gated. Rotates the server RSA key pair and returns the refreshed public_key. Operators must provide the configured operator token.

POST /api/v1/chat/completions

Creates a non-streaming chat completion. API v1 rejects stream: true.

Encrypted request body:

{
  "model": "llama-3-8b-instruct",
  "encrypted": true,
  "client_public_key": "CLIENT_PUBLIC_KEY_HERE",
  "messages": {
    "ciphertext": "ENCRYPTED_MESSAGES_PAYLOAD_HERE",
    "cipherkey": "ENCRYPTED_AES_KEY_HERE",
    "iv": "INITIALIZATION_VECTOR_HERE"
  }
}

Encrypted response shape:

{
  "encrypted": true,
  "data": {
    "encrypted": true,
    "ciphertext": "ENCRYPTED_RESPONSE_PAYLOAD_HERE",
    "cipherkey": "ENCRYPTED_AES_KEY_HERE",
    "iv": "INITIALIZATION_VECTOR_HERE"
  }
}

POST /api/v1/completions

Creates a non-streaming legacy text completion from a plaintext prompt payload. API v1 rejects stream: true. Relay-blind E2EE deployments should prefer encrypted /api/v1/chat/completions.

POST /api/v1/images/generations

Generates a deterministic local placeholder image for the supplied prompt. Optional sizing can use size (for example 512x512) or width/height.

{
  "prompt": "illustrate a relay-blind encrypted request",
  "size": "512x512",
  "seed": 42
}

GET /api/v1/health

Returns API v1 service health: status, version, service, and a current integer timestamp.

GET /api/v1/community/providers

Lists community relay and compute providers as an object: "list" response with provider records and optional metadata.

GET /api/v1/community/leaderboard

Returns the community model feedback leaderboard. The optional limit query parameter must be a positive integer.

POST /api/v1/community/contributions

Queues a community operator contribution offer and returns 202 Accepted with status: "queued" and a submission_id.

GET /api/v1/community/contributions/summary

Summarizes queued community contribution offers for maintainers.

GET /api/v1/server-providers

Lists self-hosted relay provider registry entries as an object: "list" response.

GET /api/v1/relay/server-nodes

Returns configured relay server-node diagnostics for onboarding: configured_servers, primary, secondary, and total.

POST /api/v1/relay/unregister

Operator-gated. Removes a registered compute node by server_public_key.

GET /api/v1/relay/servers/next

Returns the next registered API v1 compute-node public key for a relay-blind encrypted request.

{
  "server_public_key": "SERVER_PUBLIC_KEY_HERE"
}

POST /api/v1/relay/requests

Queues a ciphertext-only API v1 relay request envelope for the selected compute node. Plaintext fields are rejected.

{
  "server_public_key": "SERVER_PUBLIC_KEY_HERE",
  "client_public_key": "CLIENT_PUBLIC_KEY_HERE",
  "request_id": "REQUEST_ID_HERE",
  "cancel_token": "CLIENT_CANCEL_PROOF_HERE",
  "ciphertext": "ENCRYPTED_DATA_HERE",
  "cipherkey": "ENCRYPTED_AES_KEY_HERE",
  "iv": "INITIALIZATION_VECTOR_HERE"
}

POST /api/v1/relay/responses/retrieve

Retrieves a queued ciphertext response envelope by client_public_key and optional request_id.

{
  "client_public_key": "CLIENT_PUBLIC_KEY_HERE",
  "request_id": "REQUEST_ID_HERE"
}

E2EE Usage and Relay Envelope

API v1 relay traffic is mandatory end-to-end encrypted: relay endpoints are ciphertext-only and reject plaintext-like or unexpected top-level fields. Use ciphertext envelope payloads in API-facing examples.

  1. Get the server's public key from /api/v1/public-key
  2. Generate your own RSA key pair on the client
  3. Select a registered compute node with /api/v1/relay/servers/next
  4. For API v1 relay routes, send only ciphertext envelope keys
  5. Do not send plaintext top-level keys such as messages, prompt, input, content, response, or text

Internal compute-node relay control-plane routes

These routes are accounted for in the API v1 launch contract but are for registered compute nodes or relay coordination, not general user-facing API calls.

  • POST /api/v1/relay/servers/register — register or heartbeat an API v1 compute node.
  • POST /api/v1/relay/servers/unregister — unregister a compute node and cancel queued work.
  • POST /api/v1/relay/servers/poll — long-poll for the next encrypted workload.
  • POST /api/v1/relay/requests/cancel — cancel a queued or in-flight relay request with a client proof token.
  • POST /api/v1/relay/responses — store a ciphertext-only compute-node response envelope for client retrieval.

FAQ

Why token.place?

We believe in the democratization of AI. Projects like Folding@Home inspired us to leverage peer-to-peer technology for generative AI, making sophisticated language models accessible to everyone.

Is it really free?

Yes, thanks to our generous compute donors. We aim to keep it free, no strings attached, to ensure universal access to advanced AI.

How do I contribute?

You can contribute by donating compute resources or by participating in our community on GitHub. Every bit helps us grow and improve.

Learn more about our goals and how you can be a part of this initiative on our GitHub repository.

Privacy Notice

All communications are end-to-end encrypted by default and relay-mediated by design. Relay components must stay ciphertext-only (plus safe routing metadata). For maximum control, consider self-hosting via the README.
